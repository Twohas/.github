# 1인 개발자 멀티 서비스 파이프라인 (라이브)

최종 업데이트: 2026-02-25 16:22 KST
상태: 진행 중

## 1) 목표

- 1인 개발자가 번아웃 없이 여러 서비스를 개발/운영할 수 있도록 반복 가능한 파이프라인과 시스템 사이클을 구축한다.
- 서비스 수가 늘어나도 품질과 안정성을 일정하게 유지한다.

## 2) 현재 결정 사항

### 핵심 원칙
1. 표준화: 서비스별 프로젝트 구조와 개발 규칙을 동일하게 유지한다.
2. 자동화: 수동 배포와 반복 작업을 제거한다.
3. 관측성: 사용자 영향 이슈를 빠르게 감지하고 대응한다.

### 운영 모델 (3계층)
- `core-platform`: 공통 기능(auth, 로깅, 알림, 결제 어댑터)
- `service-*`: 공통 템플릿을 사용하는 독립 서비스
- `ops-repo`: IaC, CI/CD 워크플로, 모니터링 설정, 런북

### 사용자 제약 조건 (확정)
- 주력 스택: Flutter 기반 iOS/Android 앱 개발
- 운영 예정 서비스 수: 미정
- 인프라 예산: 사실상 0원 수준
- 제품 방향: 자체 서버를 운영하지 않는 서비스 우선

### 전달(Delivery) 흐름
- PR 검증: `lint -> test -> typecheck -> build`
- `main` 병합: 스테이징 자동 배포
- 릴리즈 태그: 프로덕션 배포

### 최소 운영 기준
- 서비스별 상태 확인(health check) 경로
- 구조화 로깅
- 오류 추적(error tracking)
- SLO 기반 알림(지연시간, 오류율, 결제 실패율)
- 원클릭 롤백 경로

### 모바일 우선 운영 기준 (자체 서버 없음)
- 모든 릴리즈 채널에 크래시/오류 추적 SDK 적용
- 이벤트 네이밍 규칙과 지표 오너를 고정한 분석 체계 적용
- 푸시 알림 제공자 구성 및 iOS/Android 전달 점검 절차 포함
- 배포 트랙 정책: internal -> beta -> production 순차 롤아웃

## 3) 서비스 생명주기 (System Cycle)

### Discover (주간)
- 문제, 가설, KPI를 1페이지로 정의한다.
- 주간 실험은 영향도 높은 1개만 선택한다.

### Build (일간)
- 1~2일 단위의 작은 배치로 개발한다.
- 필요 시 기능 플래그 뒤에 숨긴 상태로 배포 가능성을 유지한다.

### Release (주 1~2회)
- 릴리즈 노트를 자동 생성한다.
- 릴리즈 전에 롤백 경로를 검증한다.

### Operate (상시)
- 사용자 영향 지표를 우선 모니터링한다.
- 경량 장애 대응 템플릿으로 초기 대응 시간을 줄인다.

### Learn (주간)
- KPI와 운영 비용을 함께 검토한다.
- `유지 / 개선 / 종료` 결정을 명시적으로 수행한다.
