# 1인 개발자 멀티 서비스 파이프라인 (라이브)

최종 업데이트: 2026-02-25 16:22 KST
상태: 진행 중

## 1) 목표

- 1인 개발자가 번아웃 없이 여러 서비스를 개발/운영할 수 있도록 반복 가능한 파이프라인과 시스템 사이클을 구축한다.
- 서비스 수가 늘어나도 품질과 안정성을 일정하게 유지한다.

## 1-1) 문서 구조 및 분리 원칙

- 메인 문서는 의사결정 요약과 현재 상태만 유지한다.
- 상세 설계, 체크리스트, 템플릿은 하위 문서로 분리한다.
- 메인 문서가 길어지면 기존 상세 내용을 하위 문서로 이동하고 링크만 유지한다.

### 하위 문서 맵

- 개발 흐름 상세: `docs/solo-dev/development-flow.md`
- 앱 정의/기능 정의 상세: `docs/solo-dev/app-definition-and-feature-definition.md`

## 2) 현재 결정 사항

### 핵심 원칙

1. 표준화: 서비스별 프로젝트 구조와 개발 규칙을 동일하게 유지한다.
2. 자동화: 수동 배포와 반복 작업을 제거한다.
3. 관측성: 사용자 영향 이슈를 빠르게 감지하고 대응한다.

### 운영 모델 (3계층)

- `core-platform`: 공통 기능(auth, 로깅, 알림, 결제 어댑터)
- `service-*`: 공통 템플릿을 사용하는 독립 서비스
- `ops-repo`: IaC, CI/CD 워크플로, 모니터링 설정, 런북

### 사용자 제약 조건 (확정)

- 주력 스택: Flutter 기반 iOS/Android 앱 개발
- 운영 예정 서비스 수: 미정
- 인프라 예산: 사실상 0원 수준
- 제품 방향: 자체 서버를 운영하지 않는 서비스 우선

### 서버리스 우선 플랫폼 기준

- 기본 백엔드 모델: BaaS/서버리스 관리형 서비스 우선, 자체 호스팅 API 서버는 기본에서 제외
- 데이터 모델: 앱 로컬 저장소 기반의 오프라인 우선 구조로 시작하고, 필요한 항목만 동기화
- 서비스 분리 기준: 신규 서비스는 백엔드 복잡도를 늘리기 전에 무서버 체크리스트를 통과해야 함

### 전달(Delivery) 흐름

- PR 검증: `lint -> test -> typecheck -> build`
- `main` 병합: 스테이징 자동 배포
- 릴리즈 태그: 프로덕션 배포

### 최소 운영 기준

- 서비스별 상태 확인(health check) 경로
- 구조화 로깅
- 오류 추적(error tracking)
- SLO 기반 알림(지연시간, 오류율, 결제 실패율)
- 원클릭 롤백 경로

### 모바일 우선 운영 기준 (자체 서버 없음)

- 모든 릴리즈 채널에 크래시/오류 추적 SDK 적용
- 이벤트 네이밍 규칙과 지표 오너를 고정한 분석 체계 적용
- 푸시 알림 제공자 구성 및 iOS/Android 전달 점검 절차 포함
- 배포 트랙 정책: internal -> beta -> production 순차 롤아웃

## 3) 서비스 생명주기 (System Cycle)

### Discover (주간)

- 문제, 가설, KPI를 1페이지로 정의한다.
- 주간 실험은 영향도 높은 1개만 선택한다.

### Build (일간)

- 1~2일 단위의 작은 배치로 개발한다.
- 필요 시 기능 플래그 뒤에 숨긴 상태로 배포 가능성을 유지한다.

### Release (주 1~2회)

- 릴리즈 노트를 자동 생성한다.
- 릴리즈 전에 롤백 경로를 검증한다.

### Operate (상시)

- 사용자 영향 지표를 우선 모니터링한다.
- 경량 장애 대응 템플릿으로 초기 대응 시간을 줄인다.

### Learn (주간)

- KPI와 운영 비용을 함께 검토한다.
- `유지 / 개선 / 종료` 결정을 명시적으로 수행한다.

## 4) 30일 실행 계획

### 1주차

- 서비스 템플릿과 공통 CI 파이프라인 1개를 완성한다.
- Flutter 앱 기본값(flavor, env, 오류/크래시 SDK, 분석 SDK)을 템플릿에 포함한다.

### 2주차

- 모니터링, 알림, 오류 추적, 기본 런북을 구축한다.
- 무서버 아키텍처 체크리스트와 의사결정 게이트를 확정한다.

### 3주차

- 첫 서비스 런칭 또는 이관을 완료하고 staging/production 분리를 검증한다.
- iOS/Android 앱스토어 릴리즈 파이프라인을 점검한다.

### 4주차

- 비용 대시보드, 백업/복구 리허설, 정기 리뷰 주기를 정착시킨다.
- 1개 서비스에 대해 `유지 / 개선 / 종료` 첫 라이프사이클 리뷰를 수행한다.

## 5) 라이브 논의 로그

### 2026-02-25

- 초기 파이프라인/시스템 사이클 베이스라인을 문서화했다.
- 사용자 요청에 따라 논의 중 실시간 문서 업데이트 방식을 적용했다.
- 본 문서를 단일 기준 문서(single source of truth)로 지정했다.
- 사용자 입력(Flutter 스택, 서비스 수 미정, 인프라 예산 0원 수준, 무서버 선호)을 반영했다.
- 아키텍처 기준을 서버리스 우선 + 모바일 운영 기준으로 업데이트했다.
- 사용자 요청에 따라 문서 전체를 한글로 변환했다.
- "대략 레이아웃을 어떻게 구체화할지"에 대한 5단계 실행 로드맵과 이번 주 액션 3개를 추가했다.
- 문서 길이 관리 요청에 따라 상세 내용을 하위 문서로 분리하는 정책을 반영했다.
- 사용자가 제안한 7단계 개발 흐름과 보완 포인트를 `docs/solo-dev/development-flow.md`에 정리했다.
- 이슈 템플릿의 우선 필드 3개(AC, 분석 이벤트, QA 체크리스트)를 확정하고 상세 문서에 예시를 추가했다.
- 사용자 결정에 따라 AC를 이슈 본문 필수값에서 제외하고, 마일스톤/스프린트 + QA 체크 기준으로 업데이트했다.
- 사용자 용어에 맞춰 필드를 `분석 지표(GA/Crashlytics)`와 `테스트(테스트 방법/엣지 케이스)`로 변경했다.
- 기능 개발 단계의 종료 기준을 "단위 테스트 통과"로 확정하고, 배포 후 판단은 운영 지표 기반으로 분리했다.
- Plan에는 "넣을 것"만 관리하고 Todo에는 "나중에 검토할 것"을 두는 방식으로 확정했으며, Plan 항목 우선순위(P1/P2/P3)와 Todo->Plan 승격 규칙(주간 리뷰 한정)을 추가했다.
- 앱 정의/기능 정의를 별도 문서로 분리해 상세 논의를 시작했다(`docs/solo-dev/app-definition-and-feature-definition.md`).
- GitHub 이슈 상태 모델(`Todo -> In Plan -> In Progress -> Review -> Done`)과 상태 전이 기준, Review/Done 해석, WIP 제한(동시 2개)을 합의해 반영했다.
- 문서 위치 정정: 이슈 상태 모델 관련 합의 내용은 `docs/solo-dev/development-flow.md`에서 `docs/solo-dev/app-definition-and-feature-definition.md`로 이동했다.
- `app-definition-and-feature-definition.md`의 기능 정의 영역을 Plan/Todo 이분법 대신 이슈 상태 모델 중심 표현으로 정리했다.
- `app-definition-and-feature-definition.md`의 8/9 섹션 중복을 제거하고, 상태 모델/백로그 운영 규칙을 하나의 상위 섹션으로 통합했다.
- 문서 구조 정정: 상태 정의/전이/WIP는 `2-B 기능 정의서`로 올리고, 8번 섹션은 GitHub Issues/Projects/Milestones 중심의 도구 운영 규칙으로 재구성했다.
- 우선순위 체계를 `P0/P1/P2`에서 `P1/P2/P3`로 변경하고 해석 기준을 정리했다.
- 배포 후 버그 이슈 연동 규칙을 추가하고(`introduced_in`, `symptom_area`, `origin`, `caused_by_issue`), 규칙의 운영 이유를 함께 문서화했다.
