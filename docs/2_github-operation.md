# GitHub 이슈 및 프로젝트 운영 가이드

최종 업데이트: 2026-02-25 KST
상태: 진행 중

## 1) 목적

- 기능 개발을 GitHub 이슈 기반으로 운영하기 위한 규칙을 정의한다.
- 이슈 상태, 우선순위, 라벨, 마일스톤 관리를 표준화하여 일관된 개발 프로세스를 유지한다.

## 2) 이슈 생성 원칙

### 기본 원칙

- 기능은 1이슈 1기능 단위로 생성한다.
- 이슈 본문은 반드시 템플릿을 사용한다.
- 초기 상태는 `Todo`로 설정한다.

### 이슈 템플릿 필수 필드

- 기능명
- 초기 상태: Todo
- 우선순위: P1 | P2 | P3
- 기능 설명
- 사용자 시나리오

## 3) 이슈 상태 모델

### 상태 정의

| 상태 | 설명 |
|------|------|
| Todo | 앞으로 구현할 이슈 (백로그) |
| In Plan | 해당 마일스톤에 포함된 이슈 |
| In Progress | 실제 구현 중인 이슈 |
| Review | 구현 완료 후 리뷰/검증 중인 이슈 |
| Done | 완료 판정을 받은 이슈 (개발 완료 기준) |

### 상태 전이 기준

| 전이 | 조건 |
|------|------|
| Todo → In Plan | 마일스톤 포함 + 우선순위(P1/P2/P3) 지정 |
| In Plan → In Progress | 착수 결정 + 작업 브랜치 또는 PR 생성 |
| In Progress → Review | 단위 테스트 통과 + 이슈 체크 항목 작성 완료 |
| Review → Done | 리뷰 승인 + 검증 통과 + 마일스톤 완료 조건 충족 |

### WIP 제한

- `In Progress` 동시 진행 이슈는 최대 2개로 제한한다.
- 예외: 치명 버그, 스토어 심사 리젝 대응 건

## 4) 우선순위 규칙

### 우선순위 정의

| 우선순위 | 설명 |
|----------|------|
| P1 | 릴리즈 필수 (핵심 동작/블로커) |
| P2 | 이번 마일스톤 핵심 가치 기능 |
| P3 | 있으면 좋은 개선 (필수 아님) |

### 적용 기준

- 모든 `In Plan` 이슈는 우선순위가 반드시 지정되어야 한다.
- P1이슈는 마일스톤당 최소 1개 이상 포함을 권장한다.

## 5) GitHub Projects 연동

### 보드 운영

- 보드 컬럼은 상태 모델과 동일하게 운영한다: `Todo / In Plan / In Progress / Review / Done`
- 상태 변경은 작업 시작/리뷰 시작/완료 판정 시점에 즉시 반영한다.

### 프로젝트 설정 권장 사항

- default View: 현재 마일스톤의 In Plan/In Progress 이슈만 표시
- 백로그 View: Todo 전체 표시

## 6) GitHub Milestones 연동

### 마일스톤 운영

- `In Plan` 이슈는 반드시 하나의 마일스톤에 연결한다.
- 마일스톤 종료 시점에 `Done` 이슈만 완료 처리하고 나머지는 `Todo` 또는 다음 `In Plan`으로 재분류한다.

### 마일스톤命名 규칙

- 형식: `v{major}.{minor}.{patch}` (예: v1.0.0)
- 제목: `{서비스명} v{버전} 배포`

## 8) 사후 버그 이슈 연동 규칙

### 목적

- 배포 후 버그를 원인 미확정 상태에서도 빠르게 등록하고, 이후 추적 품질을 높인다.
- 기본 원칙: 최초 등록은 증상 기준, 원인 확인 후 링크 보강

### 필수 메타 필드

- `introduced_in`: 버그가 확인된 앱 버전 (예: v1.2.0)
- `symptom_area`: 증상 영역 (예: login, payment, sync)
- `origin`: known 또는 unknown
- `caused_by_issue`(선택): 원인 이슈 번호 (예: #123)

### 등록 규칙

- 원인 이슈를 모르면 `origin: unknown`으로 생성하고 즉시 처리 시작
- 억지 이슈 연결은 금지하고, 원인 확인 후에만 `caused_by_issue` 추가
- 원인 미확정 상태라도 `introduced_in`과 `symptom_area`는 반드시 기록

## 9) 자동화 원칙

### 자동화 대상

- 상태/우선순위 누락 이슈는 마일스톤 포함 전 보완한다.
- 새 이슈 생성 시 기본 라벨 자동 추가 권장

### 권장 GitHub Actions

- 이슈 생성 시 기본 라벨 자동 추가
- P1 이슈 생성 시 알림 설정
- 마일스톤 종료 3일 전 알림

## 10) 이슈 관리 체크리스트

- [ ] 기능 정의가 명확한가?

- [ ] 단위 테스트가 통과했는가?
