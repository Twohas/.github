# 개발 흐름 상세 (Flutter, 무서버 우선)

최종 업데이트: 2026-02-25 16:00 KST
상태: 진행 중

## 1) 사용자 제안 원안

1. 앱 정의 + 기능 정의
2. 기능 정의에 따라 GitHub 이슈 등록
3. GitHub 템플릿 레포로 프로젝트 레포 생성
4. GitHub 이슈에 따라 순차적으로 기능 개발
5. 버전에 들어갈 기능 완료 시 TestFlight 배포, Play Store Internal 배포
6. 테스트 완료 후 정식 배포
7. 앱 모니터링 및 새로운 서비스 준비(1번부터 반복)

## 2) 전체 평가지점

- 흐름 자체는 매우 좋고, 솔로 개발자에게 현실적인 루프다.
- 특히 `이슈 기반 개발 -> 내부 배포 -> 정식 배포 -> 모니터링` 순서는 유지해야 한다.
- 부족한 부분은 "진입/종료 기준"과 "출시 전 품질 게이트"가 아직 명시되지 않은 점이다.

## 3) 보완이 필요한 항목

### A. 1단계 보완: 앱/기능 정의 전에 성공 기준 고정

- 추가 권장: 기능 목록보다 먼저 `성공 KPI 3개`와 `4주 목표값`을 확정
- 이유: 기능은 늘어나기 쉽고, 지표가 없으면 배포 후 판단이 어려움

### B. 2단계 보완: 이슈 템플릿 표준화

- 추가 권장: 기능 이슈를 아래 필드로 고정
- 필수 필드: 배경, 사용자 시나리오, 분석 지표, 테스트, 릴리즈 영향도
- 이유: 구현 누락과 재작업을 줄이고, 나중에 회고 가능한 기록을 남김

### C. 4단계 보완: 개발 순서에 우선순위 규칙 삽입

- 추가 권장: 순차 개발 기준을 `사용자 가치 -> 리스크 -> 의존성` 순으로 고정
- 이유: "등록 순서"만 따라가면 중요한 기능이 늦어질 수 있음

### D. 5단계 보완: 내부 배포 전 품질 게이트 추가

- 추가 권장: Internal/TestFlight 전 필수 게이트
- 게이트: 빌드 성공, 핵심 시나리오 수동 점검, 크래시 SDK 활성 확인, 분석 지표 수집 확인
- 이유: 내부 테스트 시간을 기능 검증에 집중시키고 기초 결함 유입을 방지

### E. 6단계 보완: 정식 배포를 단계적 롤아웃으로 전환

- 추가 권장: 전면 배포 대신 staged rollout(예: 5% -> 20% -> 100%)
- 이유: 무서버 구조에서도 클라이언트 결함 영향은 크므로 점진 배포가 안전함

### F. 7단계 보완: 모니터링 후 의사결정 루프 명시

- 추가 권장: 배포 후 48시간 모니터링 + 주간 `유지/개선/종료` 회의(1인이라도 문서화)
- 이유: 다음 서비스로 넘어가기 전에 현재 서비스 상태를 닫아야 운영 부채가 누적되지 않음

## 4) 개선된 실행 플로우 (제안)

1. 앱 정의 + 기능 정의 + 성공 KPI 확정
2. 표준 이슈 템플릿으로 기능 이슈 등록
3. 템플릿 레포로 프로젝트 생성(기본 SDK/환경 포함)
4. 우선순위 규칙 기반으로 이슈 구현
5. 내부 배포 전 품질 게이트 통과
6. TestFlight / Internal 배포 후 테스트
7. staged rollout으로 정식 배포
8. 48시간 모니터링 + 주간 의사결정 후 다음 서비스 시작

## 5) 다음 논의에서 다룰 상세 항목

- GitHub 이슈 템플릿 실제 항목 정의
- 템플릿 레포 기본 구조(Flutter flavor/env/SDK)
- 내부 배포 전 품질 게이트 체크리스트
- 배포 후 모니터링 대시보드 최소 지표 세트

### 참조

- 앱 정의/기능 정의 상세 문서: `docs/solo-dev/app-definition-and-feature-definition.md`

## 6) 이슈 템플릿에 먼저 넣을 핵심 3개 필드

### 1. 사용자 시나리오(또는 기능 목적)

- 왜 필요한가: 구현 우선순위와 범위를 흔들리지 않게 유지할 수 있음
- 작성 예시:
  - 대상 사용자: 신규 가입 사용자
  - 해결하려는 문제: 로그인 진입 장벽 감소
  - 핵심 흐름: 이메일 로그인 -> 홈 진입

### 2. 분석 지표 (Google Analytics / Google Crashlytics)

- 왜 필요한가: 배포 후 사용성과 안정성을 동시에 수치로 확인할 수 있음
- 작성 예시:
  - Google Analytics: `login_attempt`, `login_success_rate`
  - Google Crashlytics: 로그인 화면 크래시 건수, 비정상 종료율
  - 지표 연결: 로그인 성공률, 크래시 프리 사용자 비율

### 3. 테스트

- 왜 필요한가: 내부 배포 전에 기능 신뢰도를 확보하고 회귀를 줄일 수 있음
- 작성 예시:
  - 테스트 방법: 단위 테스트(필수) + Android/iOS 실기기 수동 테스트(권장)
  - 엣지 케이스: 네트워크 끊김, 중복 탭, 앱 백그라운드 복귀, 오래된 앱 버전 데이터
  - 결과 기록: 실패 케이스 재현 절차와 수정 확인 링크

### 참고: 단계별 종료 기준

- 기능 개발 단계 종료 기준: 해당 이슈의 단위 테스트 통과
- 배포 후 운영 단계 판단: 실제 사용자 데이터(Analytics/Crashlytics) 기반으로 별도 판단

### 참고: 완료 상태 관리는 마일스톤/스프린트로 처리

- 본 문서는 사용자 결정에 따라 AC를 이슈 본문 필수값에서 제외한다.
- 대신 완료 판정은 GitHub 마일스톤/스프린트 상태와 단위 테스트 통과 여부를 함께 사용한다.

## 7) 이슈 정의/상태 모델 문서 위치

  `docs/solo-dev/app-definition-and-feature-definition.md`에서 관리한다.
