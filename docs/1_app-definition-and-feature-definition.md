# 앱 정의 / 기능 정의 상세

최종 업데이트: 2026-02-25 KST
상태: 논의 중

## 1) 목적

- 아이디어를 "개발 가능한 스펙"으로 전환한다.
- 기능 추가보다 우선순위와 성공 판단 기준을 먼저 고정한다.
- 1인 개발 기준으로 빠르게 만들고, 이슈 상태 기반으로 범위와 진행을 관리한다.

## 2) 1단계 산출물 구조

### A. 앱 정의서 (서비스 1페이지)

- 문제 정의: 어떤 불편을 해결하는가
- 타깃 사용자: 누구를 위한 앱인가
- 핵심 가치 제안: 기존 대안 대비 무엇이 더 나은가
- 성공 지표: 출시 4주 기준 KPI 2~3개
- 배포 채널: iOS/Android 우선순위
- 수익 모델 가설: 무료/구독/광고/인앱결제 여부

### B. 기능 정의서

- 이슈 목록: 기능을 GitHub 이슈 단위로 정의
- 상세한 이슈 관리 규칙(상태 모델, 전이 기준, 우선순위, 라벨 등)은 `docs/2_github-operation.md`에서 관리
- 의존성: 외부 서비스/SDK/정책 의존 사항
- 위험 요소: 심사 리젝 가능성, 구현 난이도, 운영 리스크

## 3) 앱 정의 체크리스트

- 한 줄 문제 정의가 20자~40자 안에서 명확한가
- 타깃 사용자가 1차/2차로 구분되어 있는가
- KPI가 행동 지표 중심인가(단순 다운로드 수 지양)
- KPI마다 측정 도구가 정해져 있는가(GA/Crashlytics 등)
- 이번 버전의 수익화 여부가 명시되어 있는가

## 4) 기능 정의 체크리스트

- 해당 마일스톤의 `In Plan` 이슈가 3~7개 사이로 제한되어 있는가
- `In Plan` 이슈의 모든 항목에 우선순위(P1/P2/P3)가 있는가
- 각 기능에 사용자 시나리오가 연결되어 있는가
- 각 기능에 분석 지표와 테스트 항목이 연결되어 있는가
- `Todo` 백로그가 지속적으로 축적/정리되는 구조인가

## 5) 기능 항목 템플릿 (초안)

다음 형식을 기능별로 반복 작성한다.

```md
- 기능명:
- 초기 상태: Todo
- 우선순위: P1 | P2 | P3
- 기능 설명:
- 사용자 시나리오:
- 분석 지표:
  - Google Analytics:
  - Google Crashlytics:
- 테스트:
  - 테스트 방법:
  - 엣지 케이스:
- 의존성/리스크:
```

## 6) 논의 순서 제안

1. 앱 한 줄 문제 정의 확정
2. 타깃 사용자 1차/2차 확정
3. KPI 2~3개 확정
4. 마일스톤 대상 이슈(`In Plan`) 3~7개 확정
5. 백로그 이슈(`Todo`) 수집
6. `In Plan` 이슈별 분석 지표/테스트 연결

## 7) 다음 회의에서 바로 채울 항목

- 앱 이름(가칭)
- 문제 정의 1문장
- 타깃 사용자 1차/2차
- 출시 4주 KPI 3개
- 마일스톤 대상 후보 이슈 목록(최대 7개)
- 백로그 후보 이슈 목록

## 8) 기능 관리 도구 운영 규칙

- 상세한 GitHub 이슈/프로젝트/마일스톤/라벨/자동화 운영 규칙은 `docs/2_github-operation.md`에서 관리한다.
