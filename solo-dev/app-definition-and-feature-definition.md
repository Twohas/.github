# 앱 정의 / 기능 정의 상세

최종 업데이트: 2026-02-25 16:22 KST
상태: 논의 중

## 1) 목적

- 아이디어를 "개발 가능한 스펙"으로 전환한다.
- 기능 추가보다 우선순위와 성공 판단 기준을 먼저 고정한다.
- 1인 개발 기준으로 빠르게 만들고, 이슈 상태 기반으로 범위와 진행을 관리한다.

## 2) 1단계 산출물 구조

### A. 앱 정의서 (서비스 1페이지)

- 문제 정의: 어떤 불편을 해결하는가
- 타깃 사용자: 누구를 위한 앱인가
- 핵심 가치 제안: 기존 대안 대비 무엇이 더 나은가
- 성공 지표: 출시 4주 기준 KPI 2~3개
- 배포 채널: iOS/Android 우선순위
- 수익 모델 가설: 무료/구독/광고/인앱결제 여부

### B. 기능 정의서

- 이슈 목록: 기능을 GitHub 이슈 단위로 정의
- 상태 모델: `Todo / In Plan / In Progress / Review / Done`
- 상태 전이 기준:
  - `Todo -> In Plan`: 마일스톤 포함 + 우선순위(P1/P2/P3) 지정
  - `In Plan -> In Progress`: 착수 결정 + 작업 브랜치 또는 PR 생성
  - `In Progress -> Review`: 단위 테스트 통과 + 이슈 체크 항목 작성 완료
  - `Review -> Done`: 리뷰 승인 + 검증 통과 + 마일스톤 완료 조건 충족
- 우선순위 규칙: `P1 / P2 / P3`
- 우선순위 해석:
  - `P1`: 릴리즈 필수(핵심 동작/블로커)
  - `P2`: 이번 마일스톤 핵심 가치 기능
  - `P3`: 있으면 좋은 개선(필수 아님)
- 작업량 제한: `In Progress` 동시 진행 최대 2개(예외: 치명 버그/심사 리젝 대응)
- 리뷰 해석: `Review`는 코드 리뷰 + 기능 검증을 함께 수행
- 완료 정의: `Done`은 개발 완료 기준(운영 성과 평가는 별도 루프에서 판단)
- 백로그 규칙: `Todo`는 백로그로 유지하고 `Todo -> In Plan` 승격은 주간 리뷰에서만 수행
- 의존성: 외부 서비스/SDK/정책 의존 사항
- 위험 요소: 심사 리젝 가능성, 구현 난이도, 운영 리스크

## 3) 앱 정의 체크리스트

- 한 줄 문제 정의가 20자~40자 안에서 명확한가
- 타깃 사용자가 1차/2차로 구분되어 있는가
- KPI가 행동 지표 중심인가(단순 다운로드 수 지양)
- KPI마다 측정 도구가 정해져 있는가(GA/Crashlytics 등)
- 이번 버전의 수익화 여부가 명시되어 있는가

## 4) 기능 정의 체크리스트

- 해당 마일스톤의 `In Plan` 이슈가 3~7개 사이로 제한되어 있는가
- `In Plan` 이슈의 모든 항목에 우선순위(P1/P2/P3)가 있는가
- 각 기능에 사용자 시나리오가 연결되어 있는가
- 각 기능에 분석 지표와 테스트 항목이 연결되어 있는가
- `Todo` 백로그가 지속적으로 축적/정리되는 구조인가

## 5) 기능 항목 템플릿 (초안)

## 6) 논의 순서 제안

1. 앱 한 줄 문제 정의 확정
2. 타깃 사용자 1차/2차 확정
3. KPI 2~3개 확정
4. 마일스톤 대상 이슈(`In Plan`) 3~7개 확정
5. 백로그 이슈(`Todo`) 수집
6. `In Plan` 이슈별 분석 지표/테스트 연결

## 8) 기능 관리 도구 운영 규칙

### GitHub Issues

- 기능은 1이슈 1기능 단위로 생성한다.
- 이슈 본문은 템플릿(사용자 시나리오/분석 지표/테스트)을 사용한다.

### GitHub Projects

- 보드 컬럼은 상태 모델과 동일하게 `Todo / In Plan / In Progress / Review / Done`으로 운영한다.
- 상태 변경은 작업 시작/리뷰 시작/완료 판정 시점에 즉시 반영한다.

### GitHub Milestones

- `In Plan` 이슈는 반드시 하나의 마일스톤에 연결한다.
- 마일스톤 종료 시점에 `Done` 이슈만 완료 처리하고 나머지는 `Todo` 또는 다음 `In Plan`으로 재분류한다.

### Labels and Automation

- 우선순위 라벨: `P1`, `P2`, `P3`
- 성격 라벨(권장): `feature`, `bug`, `tech-debt`, `store-review`
- 자동화 원칙: 상태/우선순위 누락 이슈는 마일스톤 포함 전 보완한다.

### 사후 버그 이슈 연동 규칙 (설명 포함)

- 목적: 배포 후 버그를 원인 미확정 상태에서도 빠르게 등록하고, 이후 추적 품질을 높인다.
- 기본 원칙: 최초 등록은 증상 기준, 원인 확인 후 링크 보강.

- 필수 메타 필드:
  - `introduced_in`: 버그가 확인된 앱 버전 (예: `v1.2.0`)
  - `symptom_area`: 증상 영역 (예: `login`, `payment`, `sync`)
  - `origin`: `known` 또는 `unknown`
  - `caused_by_issue`(선택): 원인 이슈 번호 (예: `#123`)

- 등록 규칙:
  - 원인 이슈를 모르면 `origin: unknown`으로 생성하고 즉시 처리 시작
  - 억지 이슈 연결은 금지하고, 원인 확인 후에만 `caused_by_issue` 추가
  - 원인 미확정 상태라도 `introduced_in`과 `symptom_area`는 반드시 기록

- 왜 이렇게 운영하나:
  - 원인 조사 대기 없이 대응을 시작할 수 있어 대응 속도가 빨라진다.
  - 버전/증상/원인 정보를 단계적으로 쌓아 나중에 추적 정확도가 올라간다.
  - 잘못된 연관 이슈 연결로 인한 관리 혼선을 줄일 수 있다.
